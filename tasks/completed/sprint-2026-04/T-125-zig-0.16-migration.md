- [x] 已识别所有std.fs.cwd()用法
- [x] 已确认迁移模式：
  - `std.fs.cwd().openFile()` → `utils.readFileAlloc()` (读取)
  - `std.fs.cwd().createFile()` → `utils.writeFile()` (写入)
  - `std.fs.cwd().openDir()` → `utils.openDir()` (目录迭代)
  - `std.fs.cwd().readFileAlloc()` → `utils.readFileAlloc()`
  - `std.fs.cwd().makeDir()` → `utils.makeDirRecursive()`
  - `std.fs.cwd().access()` → `utils.fileExists()`

### Migration Patterns
参考 `src/skills/generator.zig` 中的成功模式：
```zig
// OLD (Zig 0.15):
const file = try std.fs.cwd().openFile("path", .{});
defer file.close();
const content = try file.readToEndAlloc(allocator, max_size);

// NEW (Zig 0.16):
const content = try utils.readFileAlloc(allocator, "path", max_size);
defer allocator.free(content);
```

## Lessons Learned

### 成功经验

1. **分阶段迁移策略有效**
   - 按模块分组（harness → tools → utils → extension → workspace）
   - 每个阶段独立commit，便于问题定位和回滚
   - 实际证明：3个commits完成迁移，无需回滚

2. **Utils wrapper模式优秀**
   - 统一封装Zig 0.16 API变化
   - 使用`utils.readFileAlloc()`, `utils.writeFile()`, `utils.openDir()`等
   - 未来Zig版本升级只需修改utils层，不影响业务代码

3. **Zig 0.16关键API变化**
   - `std.fs.cwd().openDir()` 保留但需传入io参数给iterate/close
   - `std.fs.openDirAbsolute()` 已移除 → 使用`cwd().openDir(relative_path)`
   - `std.time.milliTimestamp()` 已移除 → 使用C `clock_gettime()`
   - `std.ArrayList.init(allocator)` → `.empty` + `deinit(allocator)`
   - `std.json.stringifyAlloc()` → `stringify(..., writer())`

4. **快速修复技巧**
   - 遇到API变化，先在已有代码中搜索正确用法
   - 例如：fs_helper.zig中已有Zig 0.16兼容实现
   - 批量替换时使用sed加速（如添加import）

### 遇到的挑战

1. **File vs utils wrapper选择**
   - 某些代码需要File handle（如writer）
   - 某些只需要读写内容
   - 解决：统一迁移到utils，简化File handle管理

2. **build.zig的openDirAbsolute**
   - 初期认为需要build-time代码生成
   - 实际发现项目使用auto/registry.zig手动维护
   - 简化方案：直接移除build-time扫描

3. **Error union vs Optional语法混淆**
   - `if (result) |val| else |err|` 只用于error union
   - Optional使用`if (optional) |val|`或`!= null`判断
   - 迁移时多次遇到此问题

### 适用于后续迁移

- 任何Zig版本升级都可使用此分阶段策略
- utils wrapper层已为未来变化做好准备
- 测试覆盖确保迁移安全性
