# Pay-as-you-go AI 登录与结算系统分析

**分析对象**: [Machine Payments Protocol (MPP)](https://mpp.dev/overview) + Temple Wallet 集成方案  
**分析日期**: 2026-04-05  

**核心结论**:
- **技术上完全可以实现**，而且这是一个能打出差异化的商业模式。
- **但直接用 Temple 钱包（Tezos）做主结算层有很多坑**：波动性、gas 费、用户门槛、合规风险。
- **更推荐的做法**：用 "**链下信用账户 + 链上钱包充值**" 的混合架构，或者直接用稳定币/Stripe 做充值入口。
- MPP 的 HTTP 402 + `Authorization: Payment` 模式对你的场景**参考价值有限**，真正有价值的是它的 "**无订阅、按量付费**" 理念。

---

## 1. 你想做的系统是什么

用一句话概括：
> **"一个预付钱包制的 AI 服务平台。用户先往钱包里存钱，用 AI 服务时按量扣费，余额不足就停用。"**

这和当前主流订阅制（ChatGPT Plus, Claude Pro）的核心差异：
| 维度 | 订阅制 | 你的预付/按量付费模式 |
|------|--------|----------------------|
| **付费门槛** | 固定月费 | 存多少用多少 |
| **灵活性** | 用多用少都一个价 | 精确按 token/请求计费 |
| **适合人群** | 高频用户 | 低频、间歇性用户 |
| **现金流** | 可预测 | 需要持续充值 |
| **用户心理** | "不用就亏了" | "完全可控" |

这个模式本身是**成立的**，OpenRouter 就是按量计费的成功案例。但 OpenRouter 用的是**传统支付**（信用卡充值余额），不是区块链钱包。

---

## 2. MPP 是什么，对你有没有参考价值

### 2.1 MPP 的核心机制

MPP (Machine Payments Protocol) 是一个**机器对机器支付协议**，核心思想：
1. 客户端请求资源：`GET /resource`
2. 服务端返回 **402 Payment Required**，并带 `WWW-Authenticate` header（支付挑战）
3. 客户端完成支付（签名交易、付发票等）
4. 客户端重试请求，带 `Authorization: Payment` header（支付凭证）
5. 服务端验证支付，返回资源：`200 OK` + `Payment-Verify`

### 2.2 MPP 和你的需求的关系

**直接关系不大。**

MPP 解决的是 "**机器之间如何标准化地请求-支付-交付资源**"，它是协议层的创新。而你想做的是 "**用户登录后往钱包充钱，然后在平台内部消费 AI 服务**"，这是**应用层的产品设计**。

你可以**借鉴 MPP 的理念**（无订阅、按量付费、去账户化），但不需要实现 HTTP 402 那套协议。

## 3. Temple 钱包 + Tezos 做结算的可行性分析

### 3.1 Temple 钱包是什么
- **Tezos (XTZ)** 生态的浏览器扩展钱包
- 用户需要安装 Chrome/Firefox 扩展
- 支持 XTZ 转账和 DApp 交互
- Tezos 的 gas 费很低（通常几分钱人民币），确认速度较快

### 3.2 技术架构方案

#### 方案 A：纯链上结算（不推荐）
```
用户 → Temple 签名 → 智能合约存款
                ↓
         每次调用 AI 服务时
         用户再签名一笔小额转账
                ↓
         服务端验证链上到账后返回 AI 结果
```

**问题**：
- 每调用一次 AI 就要签一次名、等一次确认 —— **UX 灾难**
- AI 调用可能是秒级的多次请求，链上交互跟不上
- 钱包弹窗频繁，用户会疯

#### 方案 B：链下信用账户 + 链上钱包充值（推荐）
```
充值流程：
用户 → Temple 签名 → 智能合约存款/授权
                ↓
         服务端监听链上事件
                ↓
         给用户链下账户加余额

消费流程：
用户调用 AI 服务 → 服务端内部扣减余额
                ↓
         余额不足 → 提示充值
                ↓
         余额充足 → 返回结果

提现流程（可选）：
用户申请提现 → 服务端审核 → 从合约/热钱包转回用户地址
```

这是**最合理的架构**。OpenRouter、很多游戏平台、云服务都采用这种 "链上充值 + 链下记账" 的混合模式。

## 4. 关键风险与应对

### 4.1 加密货币波动性风险 ⚠️ 重要

你收的是 XTZ，但 AI API（OpenAI, Anthropic）收的是 USD。如果 XTZ 价格暴跌 30%，你收的钱可能不够付 API 账单。

**应对**：
- **用稳定币（USDC/USDT）结算**，不要用 XTZ
- 如果必须用 Tezos 生态，可以用 **Tezos 上的 wrapped USDC（USDtz 或 Etherlink USDC）**
- 实时汇率换算：用户充值时按当前 XTZ/USD 汇率折算成 "USD 信用额度"

### 4.2 用户门槛风险 ⚠️ 非常重要

 Temple 钱包意味着用户必须：
1. 知道什么是加密货币
2. 安装浏览器扩展
3. 去交易所买 XTZ/USDC
4. 理解 gas 费、地址、交易确认

这直接把**99% 的潜在用户**挡在门外。

**应对**：
- **提供双轨制**：crypto 充值 + Stripe/支付宝/微信 充值
- crypto 面向 Web3 用户，法币充值面向普通用户
- 如果目标是大众市场，**法币充值应该是主入口**

### 4.3 合规与法律风险 ⚠️ 极其重要

不同国家对 "接受用户预付资金并提供服务" 有不同监管：
- 美国：可能涉及 money transmission，需要 MSB 牌照
- 欧盟：可能涉及支付服务指令 (PSD2)
- 中国：任何形式的加密支付都有政策风险

**应对**：
- 如果是小项目，可以先走 "**Token/Credit 系统**" 的法律灰色地带（用户购买的是平台积分，不是货币）
- 规模大了之后必须咨询律师
- 如果是主要面向海外用户，考虑注册 LLC 并接入合规的法币支付网关
- **绝对不要自己托管大量用户资金**，尽量让支付走第三方（Stripe, Coinbase Commerce）

### 4.4 退款与争议

如果用户充值后不满意服务，如何退款？
- 智能合约退款：需要实现复杂的退款逻辑
- 链下记账：你可以控制退款，但用户可能不信任你

**应对**：
- 明确 "余额不可提现，只能用于平台消费"（常见做法）
- 或者设置 "7 天内未消费可申请提现" 的规则
- 智能合约中预留管理员退款权限

### 4.5 Gas 费与小额支付

Tezos gas 虽然低，但充值时还是需要用户付一点 gas。如果用户只充 1 美元，gas 占比会很高。

**应对**：
- 设置最低充值金额（如 $5 或 $10）
- 考虑用 L2 或 rollup（但 Tezos 生态 L2 使用场景较少）

## 5. 推荐的产品与架构设计

### 5.1 混合充值入口

```
                    ┌─────────────────┐
                    │   用户注册/登录   │
                    │  (邮箱/钱包签名)  │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
        ┌─────────┐   ┌──────────┐   ┌──────────┐
        │ Stripe  │   │ 支付宝/微信 │   │ Temple   │
        │ 信用卡   │   │  (法币)   │   │ (Crypto) │
        └────┬────┘   └────┬─────┘   └────┬─────┘
             │             │              │
             └─────────────┴──────────────┘
                           │
                    ┌──────▼──────┐
                    │  余额系统     │
                    │ (USD 计价)   │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌─────────┐ ┌──────────┐ ┌──────────┐
        │ GPT-4o  │ │ Claude   │ │ 其他模型  │
        │ 按 token │ │ 按 token  │ │ 按请求   │
        └─────────┘ └──────────┘ └──────────┘
```

### 5.2 用户旅程

1. **注册**：邮箱 + 密码，或连接 Temple 钱包（钱包地址作为身份标识）
2. **充值**：
   - 法币用户：Stripe/支付宝扫码支付 → 到账后 USD 余额增加
   - Web3 用户：Temple 签名 → USDC/XTZ 转账到账 → 按汇率折算 USD 余额
3. **使用 AI**：
   - 每次 API 调用前检查余额
   - 调用后根据实际 token 消耗扣费
   - 余额低于阈值时弹窗提醒充值
4. **充值/停用**：
   - 余额 ≤ 0 时，服务暂停（只保留只读功能）
   - 余额充足时自动恢复

### 5.3 技术栈建议

| 组件 | 推荐方案 | 说明 |
|------|----------|------|
| **后端** | Zig / Node.js / Go | 处理 API 请求、余额扣减、身份验证 |
| **数据库** | PostgreSQL | 用户表、余额表、消费日志 |
| **支付（法币）** | Stripe / 支付宝 / 微信 | 稳定、合规、用户熟悉 |
| **支付（Crypto）** | Temple Wallet + Tezos 合约 | 面向 Web3 用户 |
| **AI 调用** | OpenRouter / 自建 Gateway | 统一接口，便于按量计费 |
| **结算** | 链下实时扣减 | 每请求扣费，延迟 < 10ms |

### 5.4 智能合约设计（Tezos/SmartPy 示例思路）

```python
class DepositContract(sp.Contract):
    def __init__(self, admin):
        self.init(
            admin=admin,
            balances=sp.big_map(tkey=sp.TAddress, tvalue=sp.TMutez)
        )

    @sp.entry_point
    def deposit(self):
        # 用户存入 XTZ，记录到 balances
        self.data.balances[sp.sender] += sp.amount

    @sp.entry_point
    def withdraw(self, amount):
        # 用户申请提现（可选）
        sp.verify(self.data.balances[sp.sender] >= amount)
        self.data.balances[sp.sender] -= amount
        sp.send(sp.sender, amount)
```

**注意**：如果是 USDC，Tezos 上的 USDC 要通过 Etherlink 或 wrapped token，合约逻辑会更复杂。建议找有经验的 Tezos 开发者审计。

## 6. 商业模式评估

### 6.1 这个模式的优势

1. **降低尝鲜门槛**：用户付 $1 就能试玩，不用绑定 $20/月订阅
2. **精准匹配需求**：低频用户不会为不用的时间付费
3. **现金流前置**：用户先付钱，你再买 API，没有垫资压力
4. **Web3 叙事吸引力**：如果用 crypto，容易吸引 DeFi/DAO 用户
5. **数据价值**：可以分析用户消费习惯，做动态定价

### 6.2 这个模式的劣势

1. **收入不稳定**：不像订阅制有可预测的 MRR
2. **用户流失快**：余额用完不充值就走了
3. **运营复杂度高**：要处理充值、提现、汇率、争议
4. **crypto 用户的绝对数量小**：如果只做 Temple 钱包，天花板很低
5. **定价透明导致价格战**：按量计费用户会对每 token 价格非常敏感

### 6.3 成功案例参考

- **OpenRouter**: 按量付费的 AI API 聚合平台，支持信用卡充值余额，非常成功
- **Replicate**: 按秒/按请求计费，预付余额制
- **AWS / GCP**: 本质上也是 "先绑卡/预付费，后按量扣款"
- **NovelAI / Midjourney**: 部分采用点数/积分制

**没有一个主流 AI 消费产品是纯 crypto 钱包预付的。** 这说明 crypto 路径虽然是可行的差异化点，但**不能是唯一路径**。

## 7. 给你的具体建议

### 如果目标是大众市场
**不要做纯 Temple 钱包。**
- 主入口必须是 **Stripe / 支付宝 / 微信**
- Temple 可以作为 "**Web3 登录选项**" 存在，增加科技感
- 结算货币用 USD 定价，用户看到的余额是 "$5.30"

### 如果目标是 Web3 小众市场
**可以做纯 Temple/钱包方案。**
- 用户群体懂 crypto，安装钱包不是障碍
- 用 USDC 结算，避免 XTZ 波动
- 强调 "self-custody"、"no KYC"、"pay-as-you-go" 的卖点
- 可以和 Tezos 生态的 DAO/DeFi 项目合作获客

### 如果目标是 KimiZ 本身（给 Agent 用）
这个模式非常有意思！
- **Agent 自己管理一个钱包余额**
- 每次调用 LLM/tool 时自动扣除
- 余额不足时 Agent 可以自动去充值（如果绑定了自动充值）
- 或者 Agent 转向使用更便宜的 model/provider

这其实就是 MPP 真正想解决的问题：机器对机器支付。如果你做的是一个 **Agent 基础设施**，那么这个模式甚至比人类用户场景更有价值。

## 8. 总结

### 能不能做？
**能做，而且是一个合理的差异化方向。**

### 最佳路径是什么？
**"链下 USD 余额 + 多入口充值（法币为主 + Temple 为辅）"**

### 最大坑是什么？
1. **纯 crypto 用户门槛太高** → 必须加法币入口
2. **XTZ 波动** → 用 USDC 或法币锚定
3. **合规风险** → 小额先跑，大了请律师
4. **每请求都链上交互** → 必须在链下结算

### 一句话建议
> **可以做一个 "先充值后消费" 的按量计费系统，但不要把它绑死在 Temple 钱包上。Temple 只是一个可选的登录/充值入口，主入口应该是 Stripe 或支付宝。真正有价值的创新不在于用哪个钱包，而在于 "无订阅、精准按量、余额自控" 的用户体验。**
